# 新对话交接总结：采集工具链与飞书入库方案

> 用途：用于开启新对话，让 Codex 接着执行项目工具清理、采集 skill 固化、飞书 CLI 入库验证。
> 当前约束：不要先看完整项目业务内容；优先围绕工具链、采集流程、飞书入库能力做收敛。

## 一、当前结论

我们之前的 IP 方向判断不能继续靠空想，需要先用小红书、抖音数据做验证。当前推荐把项目工具链收敛为三层：

1. 采集层：OpenCLI
2. 飞书入库层：官方 @larksuite/cli，也就是安装后暴露的 lark-cli
3. 兜底验证层：chrome:control-chrome

Playwright MCP 暂不作为小红书/抖音主采集方案，只适合本地页面验证、报表渲染、受控页面自动化。

## 二、已验证信息

### OpenCLI 采集能力

已实测可用：

```powershell
opencli xiaohongshu search "打工人猫" --limit 5 -f yaml --site-session persistent
opencli xiaohongshu note "<完整小红书URL含xsec_token>" -f yaml --site-session persistent
opencli xiaohongshu comments "<完整小红书URL含xsec_token>" --limit 5 -f yaml --site-session persistent
opencli douyin search "打工人猫" --limit 5 -f yaml --site-session persistent
```

小红书搜索、笔记详情、评论采集可用。抖音搜索可用，但 plays/comments/shares 等字段经常返回 0，不能直接当作真实数据，需要用 Chrome 或页面核验兜底。

### 飞书 CLI 状态

本机当前没有安装以下命令：

```powershell
lark-cli
feishu
lark
```

OpenCLI 的 external registry 里有 lark-cli，但状态是 installed: false。

注意：npm 上裸包名 lark-cli 不是官方飞书 CLI。官方包是：

```text
@larksuite/cli
```

官方包信息：

- 包名：@larksuite/cli
- 当前查到版本：1.0.53
- 描述：The official CLI for Lark/Feishu open platform
- 安装后命令：lark-cli
- 官方仓库：https://github.com/larksuite/cli

官方 README 声明支持：消息、云文档、多维表格、电子表格、云空间、日历、任务、通讯录、邮箱、审批、OKR、会议纪要、知识库、幻灯片等，并支持通用 API 调用。

## 三、推荐执行目标

新对话的目标不是继续扩散工具，而是做减法：

1. 盘点当前项目里的 tools、scripts、skills、package.json 中的工具相关内容。
2. 删除或废弃项目内可被官方 CLI 替代的临时 tool、wrapper、重复脚本。
3. 保留 OpenCLI 作为小红书/抖音结构化采集入口。
4. 安装并验证官方 @larksuite/cli。
5. 用官方 lark-cli 验证能否写入飞书多维表格或电子表格。
6. 创建一个完整的“小红书/抖音竞品数据采集 skill”。
7. 更新 orchestrator，把竞品数据采集放到 IP 判断和选题生产之前。

## 四、建议删除/废弃方向

需要新对话实际读取文件后再确认，不要盲删。初步方向：

- 删除临时测试脚本。
- 删除或废弃只做 webhook 推送、不能结构化写入飞书多维表格的本地飞书 wrapper。
- 删除未被引用、没有明确入口、无法验证的投递 tool。
- 清理 package.json 里不再需要的 scripts 和 dependencies。
- 保留真正能支撑采集、入库、skill 编排的文档和脚本。

如果遇到不确定用途的文件，先列入“待确认”，不要删除。

## 五、官方飞书 CLI 验证步骤

建议新对话按以下顺序执行：

```powershell
npx @larksuite/cli@latest install
lark-cli config init --new
lark-cli auth login --recommend
lark-cli auth status
```

注意：config init 和 auth login 可能需要用户在浏览器中完成飞书授权。

然后检查多维表格和电子表格能力：

```powershell
lark-cli base --help
lark-cli sheets --help
lark-cli schema
```

再做最小写入验证。优先验证多维表格，如果多维表格权限或参数复杂，再退一步验证电子表格 append。

## 六、竞品数据库字段建议

飞书多维表格建议字段：

- 平台
- 关键词
- 标题/描述
- 作者
- 链接
- 发布时间
- 点赞数
- 收藏数
- 评论数
- 转发/分享数
- 评论样本
- 内容属性：真实宠物 / IP人设 / 商业种草 / 教程 / 搬运 / 不相关
- IP方向标签
- 情绪/共鸣点
- 可借鉴点
- 风险点
- 采集时间
- 数据可信度
- 采集方式：OpenCLI / Chrome核验 / 手动补录

## 七、需要创建的 skill

建议新增：

```text
skills/00-竞品数据采集/SKILL.md
```

内容应包含：

- 触发场景：竞品调研、验证 IP 方向、采集小红书/抖音数据、看效果数据。
- 小红书采集命令：search、note、comments。
- 抖音采集命令：search，且标注部分互动字段不可靠。
- Chrome 兜底规则：只用于登录态、详情页视觉核验、OpenCLI 不完整字段补采。
- 字段映射规则：OpenCLI 输出如何映射到飞书字段。
- 去重规则：平台 + URL 优先，其次标题 + 作者 + 发布时间。
- 内容分类规则：区分真实宠物、IP人设、商业种草、教程、搬运、不相关。
- 评分规则：互动强度、评论情绪、IP可迁移性、内容制作难度、商业痕迹。
- 合规边界：不绕过登录/验证码，不高频采集，不读取 cookie/token，不执行点赞评论关注发布等写操作。
- 失败处理：OpenCLI 失败时用 Chrome 单条核验，不做批量浏览器抓取。

## 八、orchestrator 更新方向

把流程改成：

```text
00 竞品数据采集
01 IP方向验证
02 选题推荐
03 分镜脚本
```

关键原则：在没有采集数据之前，不把“阿饱/打工橘猫”视为已确定方向，只把它作为待验证假设。

## 十、注意事项

- 不要使用 npm 裸包 lark-cli，它不是官方飞书 CLI。
- 不要把项目内 webhook 推送脚本误当成飞书结构化入库方案。
- 不要用 Playwright 做小红书/抖音主采集。
- 不要在未验证数据前继续强化 IP 方向结论。
- 清理文件前要看引用关系和 package.json scripts。
- 发现疑似密钥、token、cookie 时必须脱敏，不输出完整值。
