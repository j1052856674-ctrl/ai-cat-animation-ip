# 内容运营编排器

## 一、解决什么问题

内容创作是一个多环节流水线——从选题到脚本到封面到发布到复盘再到策略调整。如果创作者每次都要手动描述"现在到了哪一步、需要读什么文档、依赖上一个Skill的什么输出"，效率极低且容易遗漏关键步骤。本编排器把整个链路串起来，创作者只需说"出选题"、"写脚本"、"做封面"、"复盘"，无需关心链路衔接细节。

## 二、需要读取什么

本编排器本身不读具体文件，但会根据触发词确保被调用的 Skill 正确读取其所需文件。编排器会检查依赖文件是否存在，不存在时提前提示用户。

## 三、用户需要提供什么输入

一条自然语言指令，对以下触发词做路由：

| 触发词 | 语义 | 路由目标 |
|--------|------|---------|
| "竞品调研" / "验证方向" / "采集数据" / "看效果数据" | 需要小红书/抖音样本 | `00-竞品数据采集` |
| "推荐选题" / "下一条拍什么" / "选题建议" | 需要新选题 | 先检查 `00-竞品数据采集` 是否有近期样本；不足时先路由到 `00-竞品数据采集`，充足时路由 `02-选题推荐` |
| "写脚本" / "出脚本" / "这条怎么拍" | 基于某条选题出脚本 | `03-脚本创作`（需要先有选题） |
| "做封面" / "出封面" / "封面和标题" | 基于脚本出封面 | `04-封面包装`（需要先有脚本） |
| "复盘" / "分析数据" / "看看昨天那条" | 复盘最新发布 | `05-发布复盘`（需要24h+数据） |
| "更新策略" / "调整方向" / "重新定位" | 策略检视 | `01-账号策略维护`（需要先有复盘） |

**标准全链路关键词**：
- "从选题开始" / "来一条完整的" → ② → ③ → ④（一次性走完选题到封面）

## 四、输出什么

编排器本身不输出最终产物，但负责：
1. **确认路由**：告知用户"将调用 [Skill名] 处理" + 理由
2. **前置检查**：确认所需文件/上游 Skill 输出已就绪；缺失时告知用户缺什么及如何补
3. **上下文传递**：将上游 Skill 的核心输出摘要传递给下游 Skill（如将选定的选题卡传递给脚本创作）
4. **链路追踪**：记录当前到了哪一步、下一步是什么，方便中断后恢复

## 五、什么时候使用

**始终使用**。本编排器是内容运营的入口——创作者不需要知道应该调用哪个 Skill，只需说出需求，编排器自动路由。

### 5.1 标准循环
```
周期开始
  → 00-竞品数据采集（采集小红书/抖音样本并写入飞书）
  → 01-账号策略维护（根据采集样本校准方向）
  → 02-选题推荐（出 5-10 条选题）
  → 创作者选定选题
  → 03-脚本创作（基于选定选题出大纲）
  → 04-封面包装（基于脚本出封面方案）
  → 创作者拍摄、剪辑、发布
  → 等待 24-48 小时
  → 05-发布复盘（分析数据）
  → 01-账号策略维护（根据复盘更新策略）
  → 回到 00/02（有新方向先采集，已有数据直接选题）
```

### 5.2 新人特殊流程
新账号（作品数 < 10 条）的特殊规则：
- ✅ `00-竞品数据采集`：优先启用，用真实样本验证 IP 方向，不把“阿饱/打工橘猫”等设定视为已确认结论
- ✅ `02-选题推荐`、`03-脚本创作`、`04-封面包装`：在有采集样本后正常使用，推荐覆盖 2-3 个方向测策略
- ⏸️ `05-发布复盘`：暂不启用（样本太小，结论不可靠）。改用简单数据记录
- ⏸️ `01-账号策略维护`：暂不启用（需要复盘结论才能做策略调整）。用初版策略文档

当 Codex 数据管线跑通 + 作品数 >= 10 条后，自动提示用户"可以启用复盘和策略维护了"。

### 5.3 中断恢复
如果创作者在执行到一半时说"继续"、"接着上次"或重新触发已有步骤，编排器会：
1. 读取上下文摘要（上次执行到哪一步）
2. 确认无需重新执行的前提下，从断点继续

## 六、和其他 Skill 的关系

```
┌────────────────────────────────────────────┐
│          内容运营编排器 (orchestrator)        │
│                                              │
│  ┌─────────────────────────────────────┐    │
│  │         标准循环                      │    │
│  │  ② 选题 → ③ 脚本 → ④ 封面 → 发布    │    │
│  │    ↑                          ↓      │    │
│  │  ① 策略 ←── ⑤ 复盘 ←── 数据采集     │    │
│  └─────────────────────────────────────┘    │
│                                              │
│  所有Skill输入/输出由编排器统一管理           │
└────────────────────────────────────────────┘
```

- **编排器不实现能力**：编排器只定义流程和路由，不重复实现任何 Skill 的能力。每个 Skill 是独立的能力单元
- **编排器不维护数据**：所有 content/ 下的文件由各自的 Skill 维护，编排器只负责确保调用链中文件存在
- **编排器是唯一入口**：创作者永远只和编排器交互，不直接指定"调用哪个 Skill"

## 七、注意事项

1. **路由歧义处理**：用户说的话可能同时匹配多个 Skill（如"看看数据"可能指竞品采集、发布复盘或查粉丝画像）。编排器会列出匹配项让用户二选一，不做自动推断
2. **依赖缺失不强行执行**：如用户说"写脚本"但尚未选定选题，编排器会提示"请先从最新选题推荐中选择一条"，而不是自行决定用哪条选题
3. **新人期规则硬执行**：作品数 < 10 条时，即使用户说"复盘"，编排器也不调用 `05-发布复盘`，而是告知新人期规则和替代方案（简单数据记录）
4. **状态持久化**：每次完成一个 Skill 调用后，编排器会在上下文中记录当前进度（完成了哪一步、下一个环节是什么），方便中断后恢复
5. **不替代创作者决策**：编排器串联流程，但关键决策点（选哪条选题、是否发、是否改策略）始终由创作者做出。编排器只提供数据驱动的建议
6. **文件缺失预案**：如果某个 Skill 需要的 content/ 文件不存在（如新项目尚未建立 `content/爆款拆解经验.md`），编排器会提示需要先创建该文件，并提供最小模板建议
7. **数据前置原则**：没有近期竞品采集数据时，不把 IP 方向当作已验证事实；选题推荐只能基于“待验证假设”输出，或先调用 `00-竞品数据采集`
