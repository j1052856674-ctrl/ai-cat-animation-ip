# 视频模型选型

在选择或比较短视频分析模型时，用这个参考文件。

## 选择原则

1. 看 API 支持什么输入类型，不要只看模型名或上下文长度。
2. 深拆优先整段视频输入；只有在不支持视频输入或成本太高时才采样帧。
3. 只要可能，就把视频和逐字稿一起喂进去。
4. 低成本模型做粗筛，强模型做最终报告。
5. 输出 schema 不要依赖某一家 provider。

## Provider 角色

| 角色 | 适合谁 | 输出预期 |
|---|---|---|
| MVP/免费试跑 | Gemini 的视频能力模型 | 先验证报告质量，不先假设生产成本 |
| 深度分析 | StepFun 视频能力模型 | 整段视频 + 逐字稿的深拆报告 |
| 批量筛选 | Qwen VL/Flash 视频能力模型 | 便宜一点的 hook / 冲突 / 时间轴粗拆 |
| 最终写作 | 内部网关里的长上下文文本模型 | 基于结构化视觉结论润色成报告 |

## 内部网关边界

像 `qwen3.7-max`、`qwen3.6-plus`、`deepseek-v4-pro`、`kimi-k2.6`、`glm-5.1` 以及代码模型，通常都很适合做文本整合和报告润色。但除非网关确认支持 `video_url`、上传视频，或者至少支持 image/frame 输入，否则不要把它们当视频阅读模型。

## 推荐 MVP

先从一个具备视频能力的模型开始，配严格 schema：

```text
Gemini 免费层试跑 或 StepFun 深拆
  -> 结构化 JSON
  -> 可选：内部长上下文模型润色报告
```

批量流程可以这样：

```text
Qwen VL/Flash 粗筛
  -> 挑选 Top 样本
  -> StepFun/Gemini 深拆
  -> 内部网关文本模型生成最终报告
```
