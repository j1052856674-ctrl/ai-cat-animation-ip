# 发布复盘

## 一、解决什么问题

创作者发布作品后常有"这条数据好不好？为什么好/不好？下条怎么改？"的困惑。没有复盘，经验全靠直觉累积，进步慢且不稳定。本 Skill 基于最新采集数据 + 历史趋势对比，定位每条作品的具体问题环节（选题、封面、标题、开头钩子、内容节奏、价值密度），输出可执行的优化建议，并向策略维护 Skill 传递是否需要调整大方向。

## 二、需要读取什么

- `tools/codex-automation/data/daily/` 最新日期的采集数据：播放量、完播率、互动率（点赞/评论/收藏/转发）、粉丝增长、观看时长分布
- `content/历史作品数据库.md`：历史作品数据，用于趋势对比
- `content/账号策略.md`：当前策略基线（预期表现参照）
- `content/粉丝画像.md`：粉丝特征，判断数据波动是否与受众匹配度有关

## 三、用户需要提供什么输入

- 触发词："复盘"、"分析数据"、"看看昨天那条"
- 必须输入：要复盘的作品标识（作品标题 / 发布日期 / "最新一条"）
- 可选输入：观察到的异常现象、特定想分析的维度

## 四、输出什么

### 4.1 核心数据看板
一条表格式数据摘要：播放量、完播率、互动率、涨粉数、收藏数，以及每项数据的历史分位数（高于/低于历史均值多少百分比）。

### 4.2 好/不好判定
- **综合评级**：A（爆款）/ B（优秀）/ C（正常）/ D（低于预期）/ F（翻车）
- **判定理由**：3-5 条基于数据的明确论据
- **趋势对比**：环比 7 天前同类作品、同比上个月同类作品

### 4.3 问题环节定位
按以下环节逐个诊断，判定是哪个环节出事：

| 环节 | 诊断指标 | 判定逻辑 |
|------|---------|---------|
| 选题 | 曝光量 / 播放量 | 曝光低 = 选题窄众 or 算法未推；曝光高但播放低 = 选题不吸引点击 |
| 封面+标题 | 点击率（播放/曝光） | 点击率低 = 封面或标题不够吸引 |
| 开头钩子 | 前 3 秒留存率 | 大量流失在前 3 秒 = 钩子不够猛 |
| 内容节奏 | 完播率 / 观看时长分布 | 中途大量流失 = 某段内容无聊或拖沓 |
| 内容价值 | 互动率（赞/评/收/转）| 看完但不互动 = 内容不够触动、不够有用或不够共鸣 |
| 结尾引导 | 关注转化率 / 评论率 | 看完但不关注不评论 = 结尾引导不足 |

### 4.4 策略更新建议
- **保持不变**：什么继续做
- **微调建议**：什么可以优化
- **停止做**：什么效果不好应放弃
- **新增测试**：什么新方向可以尝试

### 4.5 应同步给下游 Skill 的规则更新
例如：
- "选题推荐 Skill：标题中'XX'类关键词点击率持续低，降低此类关键词选题的优先级"
- "脚本创作 Skill：前 3 秒使用'反问式钩子'的完播率均值比'数据冲击式'高 15%，优先推荐反问式"
- "封面包装 Skill：白底黑字封面点击率均值高于彩色背景 8%，封面设计时优先白底"

## 五、什么时候使用

- 作品发布 24-48 小时后（数据基本稳定）
- 用户说"复盘"、"分析数据"、"看看昨天那条"
- 月度/周度内容规划前（需要总结性复盘）
- 标准工作流：`发布 → 等待 24-48h → 05-发布复盘 → 01-账号策略维护 → 02-选题推荐 → ...`

## 六、和其他 Skill 的关系

```
创作者发布作品（24-48h 后）
    ↓
05-发布复盘 ← tools/codex-automation/data/daily/, content/历史作品数据库.md, content/粉丝画像.md
    ↓
01-账号策略维护（接收: 复盘结论 + 策略更新建议）
    ↓
02-选题推荐（读: 更新后的策略）
```

- **上游**：`tools/codex-automation/data/daily/` 数据采集管线；`content/` 下的历史数据和策略文档
- **下游**：`01-账号策略维护` —— 复盘结论是策略调整的输入；各 Skill 也能收到"应同步规则更新"
- **职责边界**：只做分析和建议，不直接修改策略文档（那是 `01-账号策略维护` 的职责）。不负责数据采集（那是 Codex 管线的职责）

## 七、注意事项

1. **数据时效性**：发布后 24-48 小时是最佳复盘窗口。太早数据不稳定，太晚错过调整时机
2. **新人期阈限**：新账号至少积累 10 条作品数据后再启用复盘，否则样本太小、结论不可靠。前 10 条只做简单数据记录不做深度复盘
3. **数据缺失处理**：如果 `tools/codex-automation/data/daily/` 中没有最新日期的数据，应明确告知用户数据缺失，提供"手动输入数据"的替代方案，而不是凭记忆猜测
4. **归因诚实**：如果某个环节的数据不足以判定原因（如点击率低但不确定是封面问题还是标题问题），标注"不确定，建议 AB 测试"，不要强行归因
5. **策略建议分层**：区分"确定性建议"（连续 3 条以上数据支撑）和"观察性建议"（仅 1-2 条数据），避免对单条数据过度解读
6. **规则同步格式**：写给下游 Skill 的规则更新，必须用"原规则 + 新规则 + 变更理由"的格式，方便接受方理解和执行
