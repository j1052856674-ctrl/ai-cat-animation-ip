# Module 05: 发布复盘

> 作品发布 24-48h 后，基于采集数据 + 历史趋势对比，定位问题环节，输出可执行的优化建议与策略更新方向。

## Purpose

创作者发布作品后常有"这条数据好不好？为什么好/不好？下条怎么改？"的困惑。没有复盘，经验全靠直觉累积，进步慢且不稳定。本模块基于最新采集数据 + 历史趋势对比，定位每条作品的具体问题环节（选题、封面、标题、开头钩子、内容节奏、价值密度），输出可执行的优化建议，并向策略维护模块传递是否需要调整大方向。

## Inputs

| 输入 | 来源 | 必须 | 说明 |
|------|------|------|------|
| `work_identifier` | 用户指定 | 是 | 要复盘的作品标识：作品标题 / 发布日期 / `latest`（最新一条） |
| `observation` | 用户指定 | 否 | 观察到的异常现象、特定想分析的维度（如"感觉评论特别少"、"这条播放量异常高"） |
| `scope` | 用户指定 | 否 | 复盘范围：`single`（单条） / `weekly`（本周汇总） / `monthly`（月度趋势），默认 `single` |

## Reads

按以下优先级读取：

1. **`tools/codex-automation/data/daily/`**：最新日期的采集数据——播放量、完播率、互动率（点赞/评论/收藏/转发）、粉丝增长、观看时长分布。取最新日期的文件读取。
2. **`content/历史作品数据库.md`**：历史作品数据，用于趋势对比。按发布日期匹配同类作品（同平台 + 同内容类型）。
3. **`content/账号策略.md`**：当前策略基线，作为预期表现参照。
4. **`content/粉丝画像.md`**：粉丝特征（年龄/性别/地域/活跃时段），判断数据波动是否与受众匹配度有关。

## Outputs

### 5.1 核心数据看板

```yaml
data_dashboard:
  work_id: ""
  platform: ""               # 小红书 / 抖音
  publish_date: ""
  data_collected_at: ""      # 数据采集时间
  metrics:
    views: 0
    completion_rate: 0.0     # 完播率（抖音有，小红书无则 null）
    interaction_rate: 0.0    # 互动率
    likes: 0
    comments: 0
    saves: 0                 # 收藏
    shares: 0                # 转发
    new_followers: 0         # 发布后 24h 净增粉丝
    watch_duration_distribution: ""  # 观看时长分布，null 表示无数据
  vs_historical:
    percentile_views: null   # 播放量历史分位数（高于/低于历史均值 X%）
    percentile_interaction: null
    percentile_completion: null
```

### 5.2 综合评级

| 等级 | 判定条件（初始阈值） | 意义 |
|------|---------------------|------|
| `A` | 爆款指数明显超出账号平均水平（小红书 >0.15 / 抖音 >0.20） | 爆款，重点拆解 |
| `B` | 表现良好（小红书 >0.08 / 抖音 >0.12） | 优秀，可作内容方向参考 |
| `C` | 正常（小红书 0.03-0.08 / 抖音 0.05-0.12） | 正常，维持 |
| `D` | 低于预期（小红书 <0.03 / 抖音 <0.05） | 低迷，需复盘 |
| `F` | 严重翻车（数据远低于均值且无法归因于外部因素） | 翻车，应停止同类方向 |

输出含：
- `rating`: A/B/C/D/F
- `rationale`: 3-5 条基于数据的明确论据
- `trend_comparison`: 环比 7 天前同类作品 + 同比上个月同类作品

### 5.3 问题环节诊断

按以下环节逐个诊断，判定是哪个环节出事：

| 环节 | 诊断指标 | 数据源 | 判定逻辑 |
|------|---------|--------|---------|
| 选题 | 曝光量 / 播放量 | 播放量 | 曝光低 = 选题窄众或算法未推；曝光高但播放低 = 选题不吸引点击 |
| 封面+标题 | 点击率（播放/曝光） | 曝光量 + 播放量 | 点击率低 = 封面或标题不够吸引 |
| 开头钩子 | 前 3 秒留存率 | 观看时长分布 | 大量流失在前 3 秒 = 钩子不够猛 |
| 内容节奏 | 完播率 / 观看时长分布 | 完播率 | 中途大量流失 = 某段内容无聊或拖沓 |
| 内容价值 | 互动率（赞/评/收/转） | 互动率 | 看完但不互动 = 内容不够触动/不够有用/不够共鸣 |
| 结尾引导 | 关注转化率 / 评论率 | 涨粉率 + 评论率 | 看完但不关注不评论 = 结尾引导不足 |

**当诊断指标不可用时**（如小红书无曝光量/完播率/前3秒留存数据），对应环节标记 `diagnosis: not_available (field: {字段名})`，不做强行归因。

### 5.4 策略更新建议

| 类型 | 说明 | 示例 |
|------|------|------|
| `keep` | 什么继续做 | "场景代入型标题持续高CTR，继续使用" |
| `optimize` | 什么可以微调 | "发布时间 18:00 互动率低于 12:00，建议提前到 12:00" |
| `stop` | 什么效果不好应放弃 | "XX类选题连续 3 条互动率垫底，该方向暂停" |
| `test` | 什么新方向可以尝试 | "尝试评论区征集选题，验证用户需求匹配度" |

**区分确定性**：
- **确定性建议**（`confidence: high`）：连续 3 条以上数据支撑
- **观察性建议**（`confidence: low`）：仅 1-2 条数据，标注"待更多数据验证"

### 5.5 应同步给下游模块的规则更新

格式：`原规则 + 新规则 + 变更理由`

示例：
```
- target: 01-strategy / 02-topic-ideation
  field: 选题优先级
  old_rule: "开放式标题类关键词优先级 P1"
  new_rule: "开放式标题类关键词连续 5 条互动率 <0.03，降低优先级到 P3"
  reason: "最近 5 条该类选题互动率均值为 0.026，显著低于账号均值 0.045"

- target: 03-script
  field: 钩子类型推荐
  old_rule: "鼓励尝试多种钩子"
  new_rule: "反问式钩子的 3 秒留存率均值比数据冲击式高 15%，优先推荐反问式"
  reason: "近 10 条作品对比数据"
```

## Procedure

```
Step 1: 根据 work_identifier 定位要复盘的作品 → 确定平台、日期
Step 2: 检查数据源可用性
    ├─ tools/codex-automation/data/daily/ 存在最新日期数据？ → 读取指标
    └─ 不存在？ → 进入 Missing Input Handling（降级为手动输入模式）
Step 3: 读取 content/历史作品数据库.md → 匹配历史作品，计算分位数值
Step 4: 读取 content/账号策略.md + content/粉丝画像.md → 获取策略基线和粉丝特征
Step 5: 判断作品数量
    ├─ works_count < 10 → 降级为简单数据记录模式（不做深度归因）
    └─ works_count >= 10 → 正常复盘模式
Step 6: 计算爆款指数 → 得出综合评级（A-F）
Step 7: 逐环节诊断 → 能诊断的诊断，不能的标注 not_available
Step 8: 产出策略更新建议 → 区分确定性 vs 观察性
Step 9: 产出同步规则 → 标注 target 和 change_log
Step 10: 输出复盘报告 → result contract
```

## Missing Input Handling

| 缺失项 | 处理方式 | 降级后能否继续 |
|--------|----------|---------------|
| `tools/codex-automation/data/daily/` 无最新数据 | **不卡死**。提示用户"数据缺失"，提供手动输入替代方案——让用户手动提供播放量/互动数/涨粉数等核心指标。`diagnosis` 中标注 `data_source: manual_input` | 是（降级为手动输入） |
| 作品总数 < 10 条 | **触发新人期规则**：降级为简单数据记录模式。只做数据看板（填写可用字段），不做深度归因（不做趋势对比、不做分位数计算、不做策略更新建议）。在 report 中标注 `mode: simple_recording (works_count={n}<10)` | 是（降级为简单记录） |
| `content/历史作品数据库.md` 不存在或为空 | 无法做趋势对比和分位数计算。标注 `trend_unavailable`，只在当前单条数据基础上做可实现的分析。策略建议只输出观察性建议（`confidence: low`） | 是（深度受限） |
| `content/账号策略.md` 不存在 | 无法对比策略基线。降级为纯数据描述，不做"符合/偏离策略"的判断。标注 `missing_strategy_baseline` | 是（判断范围缩减） |
| `content/粉丝画像.md` 不存在 | 不做受众匹配度判断。标注 `missing_persona_data` | 是（判断范围缩减） |
| 特定数据字段不可用 | 各平台数据字段不同（如小红书无完播率/无前3秒留存/无曝光量）。按字段逐个标记为 `not_available`，不在对应环节做判断 | 是（逐个字段降级） |
| 数据管道完全未配置 | 引导用户：① 先配置 Codex 数据采集管线；② 或当前仅使用手动输入模式。首次使用不解锁 full mode | 是（手动输入模式） |

### 数据字段可用性矩阵

| 字段 | 小红书 | 抖音 | 缺失时标记 |
|------|--------|------|-----------|
| 播放量 | 可见（阅读量） | 可见 | — |
| 点赞/评论/收藏/分享 | 可见 | 可见 | — |
| 涨粉数 | 需手动记录 | 需手动记录 | `not_available: new_followers` |
| 完播率 | 不提供 | 可见 | `not_available: completion_rate`（仅小红书） |
| 曝光量 | 不提供 | 可见（部分） | `not_available: exposure`（仅小红书） |
| 前 3 秒留存 | 不提供 | 可见（部分） | `not_available: first_3s_retention` |
| 观看时长分布 | 不提供 | 可见（部分） | `not_available: watch_duration_distribution` |

## Result Contract

```yaml
result:
  skill_run_id: "content-ops-module-05-{YYYYMMDD}-{NNN}"
  status: completed | degraded | blocked | needs_human
  outputs:
    - review_report         # 完整复盘报告（含数据看板、评级、诊断、策略建议、同步规则）
  next_step:
    - if_rating_ge_C: "策略确认，继续发布"
    - if_rating_D_or_F: "建议触发 01-账号策略维护 做策略检视"
  blocking_reason: null     # 当 status=blocked 时填写
  warnings: []              # 降级时填写
  mode: full | simple_recording  # 正常模式或简单记录模式
```

**触发 downstream**：报告完成 → 若评级为 D/F 或用户说"更新策略"，编排器自动路由到 `01-账号策略维护`。

## Safety / Boundaries

1. **数据时效性**：发布后 24-48 小时是最佳复盘窗口。太早数据不稳定，太晚错过调整时机。如果作品发布时间 < 24h，标注 `warning: early_review (data_may_be_unstable)`。
2. **新人期阈值（硬约束）**：作品数 < 10 条时，不做深度归因——结论不可靠是在误导创作者。只做简单数据记录。即使作品表现极好或极差，也不做"策略级"判断。`mode` 字段必须为 `simple_recording`。
3. **数据缺失≠归因自由**：某个字段不可用时，对应诊断环节标注 `not_available`，不做强行归因。绝不说"应该是封面问题"当曝光量数据不可得时。
4. **归因诚实**：如果某个环节的数据不足以判定原因（如点击率低但不确定是封面问题还是标题问题），标注"不确定，建议 AB 测试"，不强行归因到单一环节。
5. **策略建议分层**：区分"确定性建议"（连续 3 条以上数据支撑）和"观察性建议"（仅 1-2 条数据），避免对单条数据过度解读。观察性建议必须标注 `confidence: low`。
6. **规则同步格式强制**：写给下游模块的规则更新必须用"原规则 + 新规则 + 变更理由"的格式，方便接受方理解和执行。缺少任一字段的同步规则视为无效。
7. **不做策略修改**：复盘只出建议，不直接修改 `content/账号策略.md`。策略文档的修改是 `01-账号策略维护` 的职责。边界清晰。
8. **不负责数据采集**：本模块不抓取数据、不调 API、不爬平台。数据采集是 `tools/codex-automation` 管线的职责。数据管线不可用时，走手动输入降级路径。
9. **爆款指数阈值是假设值**：当前阈值（小红书 0.15 / 抖音 0.20）基于赛道通用经验假设，需积累 20+ 条作品数据后重新校准。在 report 中标注 `threshold_source: assumed (require_20+_works_to_calibrate)`。
